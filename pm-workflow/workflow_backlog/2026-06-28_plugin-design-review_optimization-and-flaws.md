## 二、当前工作流可优化点及方案（Part A · CLAUDE.md 瘦身）

> 当前模式下 CLAUDE.md 由 harness 作为 project instructions **自动常驻**——它**已经是**那个「最小常驻锚」，只是不小。瘦身**不需要** orchestrator.md + SessionStart 注入那套插件管道；当前等价动作就是「把 CLAUDE.md 自己改瘦」。

实测口径：

| 段 | 行数 | 性质 |
|---|---|---|
| 工作流架构 | **203** | 四阶段详细循环 + 调整意见第零~四步 SOP —— 下沉主力 |
| 斜杠命令 | 54 | 含 changeRequest vs 调整意见 对照表（⛔ 不可下沉，见 A-3）|
| 目录结构 | 52 | 多为静态参考 |
| Agent 调用规范 | 43 | 多为必须常驻（读文件边界 / 派发自检）|
| 文件命名规范 | 40 | SemVer 全文，触发才需 |
| 会话管理建议 | 30 | 面向人类的 /clear 建议 |

另：CLAUDE.md 含 **44 处机构记忆型括注**（议题#/SSOT#/审计#/NB-/日期/实证/commit/回滚/记账）。

### A-1 外迁机构记忆（Tier 1 · 先做 · 低风险但非零）

| 项 | 内容 |
|---|---|
| **问题** | 44 处机构记忆型括注每会话常驻、对编排器**运行期**零调度价值却耗 token |
| **主论据** | 运行期 token 成本：编排器调度不需要这些溯源，却每回合常驻 |
| **辅助论据（注意范围）** | 形态同 SSOT #79 在 L1 产物禁的「内联变更标记」。但 **#79 严格只约束 L1 产物正文，CLAUDE.md 是 L2 规则文件、不在 #79 射程**——「CLAUDE.md 违反 #79」是**类比、非定性**，不能当硬论据。L2 规则文件**合法携带溯源**（维护者 / `/retro` 追根因要用，框架自我演化机制依赖它）|
| **方案** | ① 用 `strip_inline_change_markers.py`（扩扫 CLAUDE.md）生成逐条清单；② **先分类**：纯溯源括注（议题#/日期/commit）→ 外迁；**操作型括注**（含豁免 / 限定 / 例外条件，如「除非…可豁免」）→ **留**；③ 外迁项搬 `nb_log.md` / `CHANGELOG_L2.md`，原处**保留 rule→rationale 指针**（不可裸删，否则斩断 L2 演化溯源链）；④ 删除按 #79 分工**人工定夺** |
| **风险** | **低但非零**：操作型括注误删 = 规则回归；裸删溯源 = 斩断维护审计链。「人工甄别 + 保留指针」是前置条件，**不是「纯减法零回归」** |

### A-2 下沉触发门控详情（Tier 2 · 比预想小 · 逐条验）

**判定线（两条，缺一不可）**：① 无命令触发时不会自己 fire；② **不需要跨 `/clear` 存活**。

> **关键约束（2026-06-28 二次自审新增，原版漏了第②条）**：框架**推荐 stage 间 `/clear`**（会话管理建议：「每次 /nextStage 通过前」「阶段4 拼装后」），且重启标准指令**只重读 `state.md` + `decisions_ledger.md`，不重读 command 文件**。CLAUDE.md 的不可替代性正在于**它 `/clear` 后由 harness 自动重注入，command 文件内容则被清空且不再加载**。→ **凡需跨 `/clear` 存活的规则，移出 CLAUDE.md = `/clear` 后丢失**。

| ✅ 可下沉（同时满足两条判定线）| 去处 | 为何安全 |
|---|---|---|
| SemVer 版本号全文（40 行）| `nextStage.md` / `changeRequest.md` | 仅在 `/nextStage`/`/changeRequest` **当次**触发用，用完即弃、不需跨 `/clear` |
| 会话管理 /clear 建议（30 行）| README / `projectStatus.md` | 面向人类的 meta-建议，非编排器运行规则 |

| ⛔ 不可下沉（需跨 `/clear` 存活）| 原因 |
|---|---|
| **四阶段详细执行循环（`## 工作流架构` 203 行）** | 跨多个 dispatch 回合 + 跨 stage 间 `/clear` 持续需要；移出 = `/clear` 后编排器丢执行规则，恢复指令又不重读 command。**原判「下沉主力」错误 —— 它是常驻主力,只能原地压缩** |
| 调整意见第一~四步 SOP | 用户随时可在任意 stage 提调整 → 无固定命令触发 + 需跨 `/clear` |

方案：可下沉项在 CLAUDE.md 留「一句话 + 指针」，正文移入命令文件 + 同步 `ssot_anchors.md`；不可下沉项**只做原地压缩**（去冗余 / 合并重复），不搬走。

### A-3 必须常驻清单（⛔ 不能下沉，仅可原地压缩）

满足**任一**即必须常驻：(a) 无单一命令触发、随时可 fire；(b) **需跨 `/clear` 存活**（CLAUDE.md 自动重注入，command 文件不会）。

- 编排器读文件边界
- 派发 prompt 自检 3 项
- 调整意见第零步（接收下游反馈前先 AskUserQuestion）
- L1/L2 二元判定速查
- **changeRequest vs 调整意见 判定对照表** ← 守「调用 /changeRequest **之前**」的误判，下沉 = 误判已发生
- **四阶段详细执行循环（203 行）** ← 2026-06-28 自审修正：原 A-2 误列「可下沉主力」，实为跨 `/clear` 常驻（判据 b）

> 结论：插件文档 §5.1「三分法桶」**不能照抄**（把 changeRequest 对照表划进「可下沉」会把守卫搬离案发现场）；且**仅用「无命令触发」单条判据不够，必须叠加「跨 `/clear` 存活」第二条**——这是本方案首版自己踩的同型坑。

### A-4 执行纪律

- **目标区间（2026-06-28 自审下修）**：原估 −30~40% **偏乐观**——体量第一的 203 行四阶段循环须跨 `/clear` 常驻、不可外迁。**仅靠 A-1 外迁 + A-2 安全下沉（SemVer/会话管理）现实约 −15~25%**。
- **要更大降幅 → 须引入 `/clear`-存活的注入机制**（**修正原「不引入第二注入源」**）：现有 `session_start_state_summary.py` 这个 SessionStart hook（matcher `*`）**在 `/clear` 后会重新 fire**，可扩展为「注入常驻锚 / 下沉内容指针」——这是**当前工作流就能用、非插件专属**的机制。原 A-4 把「插件 orchestrator.md」与「任何 SessionStart 注入」混为一谈是错的。**真正要避免的是「两份并行权威大脑」（双 SSOT）,不是注入本身**：注入内容若只是「指针 + 速查、真源仍在 CLAUDE.md/rules」则不构成双源。
- **走流程**：高 blast-radius L2 重构（CLAUDE.md 是多个 SSOT 锚真源）→ 必须 `workflow-evolution` skill + `ssot_anchors.md` 5 要素同步 + pytest 全绿。
- **顺序**：先 A-1（仍是最低风险，但按 A-1 新分类甄别操作型括注）→ 量实减 → 再评估是否需 SessionStart 注入机制承接 A-2 的更大降幅。
