# Agent 执行方法论 v1.0

> **适用对象**：任意 Agent——本方法论不绑定具体角色（如 PM / Supervisor）、不绑定具体业务工作流。任何执行多步认知任务、需要可恢复 / 可审计 / 可协作的 Agent 均可适配。
> **读取时机**：subagent 开工前由角色规范指向本文件自行 Read。
> **核心理念**：定义任意 Agent 在执行多步任务时必须遵守的 11 条不变式（10 条主条目 + 1 条 X1 配套切分判据），按"任务时序（6 条） + 贯穿纪律（4 条）"两层组织，辅以 2 条元规则。
> **分层原则**：本文件**只定义不变式与参数槽位**，不定义具体角色 / 工作流的实例值。各 Agent 的参数实例（编号前缀、上报路径、atomic step 切分清单等）声明于配套的**参数声明文件**（如 `agent_parameters.md`），与本文件分离维护。
> **不变性约束**：本文件的不变式条目**不得修改**；扩展只能在参数层（即配套的参数声明文件）。

| 标签 | 含义 |
|------|------|

## 二、任务时序（6 条）

### T1 上下文加载

**[Must] 不变式**：开工第一步必须 Read 所有派发路径清单中的文件。禁止基于路径、文件名、或记忆中的旧版内容猜测文件内容；禁止在未读完派发清单的情况下开始后续任何时序条目。

**参数化维度**：
- 必读路径清单的来源（编排器派发 prompt）
- 读取异常的处理（如路径不存在 → 立即走 X2 阻塞上报）
- **子步骤切分（atomic step 形式）**：本节奏切分粒度（如 `t1.1` / `t1.N` 等）由各角色在 `agent_parameters.md §三 角色 atomic step 声明表` 中按 §三·X1 配套通用切分判据自决；本文件不预设条目

**不变式原因**：本工作流的 token 控制策略是"路径传递而非内容传递"。任何 Agent 破坏此条，整个 context 控制架构即崩溃。

---

### T2 初步分析

**[Must] 不变式**：读完派发文件后、进入 T3 任务探索前，必须完成一次粗粒度分析，产出**任务理解摘要**，包含：
1. 本次任务的目标与范围
2. 输入文件的内容概览
3. 初步边界判断（什么在范围内、什么不在）
4. 用于支撑 T3 探索的基础假设集合

**参数化维度**：
- 初步分析的框架
- 摘要写入位置（进度文件「任务理解摘要」节）
- 允许的初步假设粒度（避免提前做过强的方向性承诺）
- **子步骤切分（atomic step 形式）**：本节奏切分粒度（如 `t2.1` / `t2.N` 等）由各角色在 `agent_parameters.md §三 角色 atomic step 声明表` 中按 §三·X1 配套通用切分判据自决；本文件不预设条目

**不变式原因**：T3 探索的本质是"基于已有认知找缺口"。没有初步认知，探索会退化为无靶心的发散——Agent 漫无目的地质疑一切，既低效又遗漏真正关键的未知点。初步分析摘要同时是中断恢复时最快的认知重启点。

---

### T3 任务探索

**[Must] 不变式**：T2 后、T4 深入分析前（或 T4 回溯触发时），必须完成一次面向未知的探索，产出**疑问清单 + 风险清单**。探索动作包括：
1. 对任务边界是否清晰的主动质疑
2. 对输入的内部矛盾 / 缺项 / 歧义的识别
3. 对极端场景和边缘情况的穷举式发散

**参数化维度**：
- 探索框架（各角色可复用已有分析框架文件——本工作流将分析框架以 markdown 形式置于 `pm-workflow/skills/<框架名>/SKILL.md` + `template.md`，通过 Read + 按 schema 产出中间产物的方式应用，不依赖 Claude Code 平台的 Skill 工具——或自定义发散方法）
- 疑问清单的写入位置
- 探索产出与 X2 的衔接（阻塞疑问立即走 X2 暂停；非阻塞流入 T4 作为"需假设填充的空位"）
- **子步骤切分（atomic step 形式）**：本节奏切分粒度（如 `t3.1` / `t3.N` 等）由各角色在 `agent_parameters.md §三 角色 atomic step 声明表` 中按 §三·X1 配套通用切分判据自决；本文件不预设条目

**不变式原因**：未经探索即进入结构化分析，等于基于"未经质疑的前提"建塔。越到后期发现的 unknown，rework 成本越高。探索是把认知风险前置拦截。

---

### T4 深入分析

**[Must] 不变式**：T3 探索后、T5 任务执行前，必须完成一次结构化分析，产出：
1. 方案收敛结论
2. atomic step 清单（可直接灌入 T5 进度文件，**每个 step 必须满足 X1 配套切分判据**）
3. 每个 step 的验收点（作为 X3 逐步自检的判据来源）
4. 探索摘要（作为中断恢复的认知快照）

**[Must] 收敛判定**（决定 T4 是否可退出进入 T5）：
- **首要判据（定性）**：剩余 unknowns 均为非阻塞性（可用合理假设推进并记录）。非阻塞不需消除，只需挂账。
- **兜底判据（定量）**：T3 ↔ T4 迭代累计不得超过 **3 轮**。第 3 轮结束仍有 unknown **无法明确归类为阻塞 / 非阻塞**（即 Agent 不知道该把它视作阻塞还是非阻塞处理，而非"已知阻塞但无法解决"）时，**升级为元问题**走 X2 阻塞上报。已知为阻塞但无法自主解决的，直接走 X2 阻塞路径，不必包装为元问题。
- 上述兜底判据是元规则「迭代熔断」在 T3↔T4 路径上的具体应用，详见 §四·迭代熔断。

**参数化维度**：
- atomic step 的语义前缀
- 验收点格式
- 探索摘要写入位置
- **子步骤切分（atomic step 形式）**：本节奏切分粒度（如 `t4.1` / `t4.N` 等）由各角色在 `agent_parameters.md §三 角色 atomic step 声明表` 中按 §三·X1 配套通用切分判据自决；本文件不预设条目（注：T4 产出物本身是 T5 的 W1-Wn atomic step 清单，与本节奏自身的 t4.M 切分是两层概念）

**不变式原因**：分析建立对"已知"的逻辑掌控。产出的 atomic step 清单是 T5 和 X1 的前置——没有清晰的步骤拆解，原子推进无从谈起。双重兜底防止 T3↔T4 无限循环。

---

### T5 任务执行

**[Must] 不变式**：严格按 T4 产出的 atomic step 清单逐步推进。任何执行中发现的"计划与现实不符"（原计划假设失效、出现未列入的步骤、发现新依赖），**必须回到 T2/T3/T4 更新计划后再继续**，禁止在不更新计划的情况下自由发挥或"就地变通"。

**偏离回溯触发条件**：
- 执行中发现某 step 依赖的前置步骤 T4 未规划 → 回 T4 重拆
- 执行中发现新的 unknown → **先走 X2 分级处理**（不得直接回溯）；仅当 X2 解答到达且解答内容更新了对前置探索的理解时，再按解答影响范围回溯到对应条目（T3 补探索 / T4 重拆 / T2 重审）
- 执行中发现 T2 的初步理解有误（通常由 X2 阻塞解答揭示）→ 回 T2 重建
- 执行中发现 T4 计划本身存在结构性问题 → 回 T4
- 连续 3 轮回溯仍无法稳定推进 → 走元规则「迭代熔断」

> **与 X2 的衔接**：T5 偏离回溯的"发现 unknown"不是直接触发回溯，而是先走 X2 分级——非阻塞挂账继续无需回溯（避免为可挂账的细节重新探索浪费），仅阻塞经外部解答后才视情况回溯。这保持了 X2「全时序触发」原则。

**参数化维度**：
- 主体步骤的语义（起草 / 核查 / 实现 / 测试）
- 偏离计划的触发判据（角色自定义）
- 回溯的最小动作粒度
- **子步骤切分（atomic step 形式）**：本节奏 W1-Wn 清单由 T4 动态产出 + 各角色在 `agent_parameters.md §三 角色 atomic step 声明表` 中按 §三·X1 配套通用切分判据自决（不同子角色 W 步前缀不同，如 PM 阶段 4 用 TP / FA / PI / S / P / QA 等）；本文件不预设条目

**不变式原因**：Agent 最常见的质量失败不是"不会做"，而是"做着做着偏离了自己写下来的计划"。允许不更新计划的偏离 → T4 清单失去约束力 → X1 原子推进变成"按当下感觉打勾" → 质量闭环崩塌。

---

### T6 整体自检

**[Must] 不变式**：所有 atomic step 完成后、产物交付前，必须完成一次**整体层面**的自检，验证跨步属性：
1. 跨步一致性（编号、命名、引用在全局统一）
2. 完整性（任务理解摘要中承诺的范围是否全部覆盖）
3. 连贯性（叙事、逻辑、流程在整体层面成立）
4. 非阻塞问题清单齐备（所有 X2 非阻塞记录已列入成果文件相应章节）
5. **方向忠实度**（实际产出的方案方向与 T4 收敛结论一致；若有偏离，须能在进度文件中追溯到对应的回溯记录或 X4 反馈处置记录——禁止"无记录的隐式偏离"）

**与 X3 的分工**：X3 验证"每一步本身正确"，T6 验证"整体产出的跨步属性"——两者粒度和对象都不同，不重叠。

**失败处理**（按问题属性组织，与角色具体 atomic step 编号无关——具体角色的 atomic step 失败如何对应到下列属性，由参数声明文件 §角色 atomic step 声明的注释或反向映射给出）：

- **跨步一致性问题**（编号 / 命名 / 引用错位）→ 回 T5 修正涉及的 atomic step
- **完整性问题**（任务理解摘要范围未覆盖）→ 回 T4 增补 atomic step 清单；若是范围本身有误（T2 初步理解出问题）→ 回 T2 重审
- **连贯性问题**（叙事 / 逻辑 / 流程断链）→ 回 T3 补探索 + T4 重拆
- **非阻塞清单缺失**（X2 非阻塞挂账未落到位）→ 回 X2 流程，对漏挂账的不确定性补走 X2 非阻塞路径
- **方向忠实度问题**（实际产出与 T4 收敛结论不一致且无回溯记录）→ 走 X2 阻塞向需求方确认是接受新方向（更新 T4 结论）还是回 T4 强制纠偏
- **交付动作失败**（写入标记 / 外部状态更新等机械动作失败）→ 走 X3 失败处理
- 任意失败属性连续 3 轮回溯仍无法收敛 → 走元规则「迭代熔断」

**参数化维度**：
- 整体层面检查清单来源
- 交接标记文字
- 外部状态文件路径与更新字段
- **子步骤切分（atomic step 形式）**：本节奏切分粒度（如 `t6.1` / `t6.N` 等）由各角色在 `agent_parameters.md §三 角色 atomic step 声明表` 中按 §三·X1 配套通用切分判据自决；本文件不预设条目

**不变式原因**：单步自检 X3 只能验证 step-level 正确性，整体产出的跨步一致性（如"全局编号在多处引用是否一致"）必须从全局视角验证。X3 做好了而 T6 省略 → 可能产出"每块砖都对但楼歪了"的产物。

## 三、贯穿纪律（4 条）

### X1 原子化推进

> **术语定义**：本方法论中 **atomic step** 指能独立验收、独立重做、有明确产出交接的最小执行单位。

**[Must] 不变式**：任务的每一个 atomic step（涵盖 T1-T6 内部的所有 atomic step）必须写入外部进度文件（格式 `[ ]`）。每一个 atomic step 完成时立即改为 `[x]`，禁止批量更新。中断恢复时从第一个 `[ ]` 继续，已标 `[x]` 的不重复执行。

**配套要求**：
- 每个 atomic step 在进度文件中必须是独立一行
- 强耦合动作的处理方式见下方「X1 配套 · 通用切分判据」判据 2「可独立重做」——强耦合的动作应作为单一 atomic step 不予切分，而非切了再统一标 `[x]`

**参数化维度**：
- 进度文件路径
- 主体 step 的语义前缀

**不变式原因**：这是 context rot 和中断恢复的唯一解。外部持久化 + 原子性是通用状态管理原语，与"做什么"无关。与 X3 共生——X1 管"完成落盘"，X3 管"落盘前验证"，两者共同定义"真正完成一步"的语义。

#### X1 配套 · 通用切分判据

**[Must] 不变式**：T1-T6 的每一个时序条目在进入执行前，必须由角色在配套的**参数声明文件**（如 `agent_parameters.md`）的角色 atomic step 声明表中声明其切分清单；切分必须同时满足下列四条判据。禁止以"整体作为一步"的方式跳过切分，禁止在执行中临时拼凑未声明的 atomic step，禁止事后补登切分清单以倒推已完成的工作。

> 本条是 X1 的配套条款。X1 规定"每个 atomic step 必须落盘"，本条规定"何为合法的 atomic step"。切分的**动作**在参数矩阵层面完成，切分的**合规性**由本条约束。

**切分判据**：

| 判据 | 客观要求 |
|------|---------|
| 1. 可独立验收 | 每个 atomic step 须附带预设验收点，X3 可在不依赖后续 atomic step 的情况下判定其通过 / 失败 |
| 2. 可独立重做 | 单个 atomic step 失败时，重做范围仅限该 atomic step 自身；若失败必然牵连前序已 `[x]` 的 atomic step，说明切分过细或边界选错 |
| 3. 产出可追溯 | 相邻 atomic step 之间必须存在可识别的产出交接（文本段落、结论、清单条目、判定结果等），后一步的输入能明确指向前一步的产出 |
| 4. 边界贴合不确定性 | 单个 atomic step 执行过程中若必然触发 X2 阻塞（而非偶发），则切分错误——可预期的阻塞点必须作为 atomic step 的边界，而非藏在 atomic step 内部 |

**判据核心**：问"这一步失败时，我能否只重做它自己？能否独立判定它是否完成？它的产出能否交给下一步？"——三问均为是，切分合规。

**典型反模式（对照判据自检）**：

| 反模式 | 违反判据 | 表现 |
|--------|---------|------|
| 过粗合并 | 1 + 3 | 把多个独立产出合并为一个 atomic step（如"完成阶段一"作为单一 step），X3 无法独立判定，相邻交接消失 |
| 过细拆分 | 3 | 把无独立产出的子动作拆出（如"打开文件 / 阅读第一段 / 阅读第二段"），相邻 atomic step 间无可识别产出交接 |
| 耦合误拆 | 2 | 把强耦合动作拆为多个 atomic step——判据 2 失效的典型形式。例：「创建 spec / 删除骨架 placeholder」必须连续完成才保证一致，若强行拆为两步，则任何一步失败都需撤回另一步 |
| 阻塞内嵌 | 4 | 可预期的阻塞点（如"等待用户裁决"、"工具校验失败必触发"）藏在 atomic step 内部而非作为边界，导致单步执行中段被迫触发 X2 |

**与现有纪律的对接**：
- **X1**：atomic step 是 `[ ]` / `[x]` 的最小单位，切分合规才能保证"真正完成一步"的语义
- **X3**：验收点定义在 atomic step 粒度上，一个 `[x]` 对应一次且仅一次验收
- **回溯总则**：atomic step 是回溯的最小单位，回溯的最小动作是"撤回最近一个 `[x]`"
- **迭代熔断**：atomic step 是回溯轨迹的最小记录单位（即「最近撤回了哪个 `[x]`」可被精确定位），但熔断计数仍以条目级整轮回溯为单位（详见元规则「迭代熔断」）

**参数化维度**：
- 各 T 条目的 atomic step 清单（由角色在参数矩阵声明，条目内容与数量完全角色自决）
- 各 atomic step 的验收点（继承 T4 定义的验收点格式）
- 切分粒度的具体选择（角色依据业务形态自决，但须通过四条判据检验）

**不变式原因**：

不定义切分合规性，X1 / X3 的纪律在 T1-T4 前置阶段会失效——典型表现：T2 整段做完才标一个 `[x]`，T3 探索清单一次性落盘而无中间验收，T4 分析过程无法中断恢复。前置阶段本是质量杠杆最大的环节，却成为纪律覆盖最薄的环节，整套方法论的原子化承诺因此出现结构性漏洞。

同时，不给具体切分表而给判据，是对"方法论管不变式、参数矩阵管角色差异"分层原则的严格遵守——PM 的 T2 切四步合理、Supervisor 的 T2 切两步也合理、未来测试 Agent 的 T2 切六步同样合理，方法论不对业务形态做任何预设，只保证不论怎么切，落盘、验收、回溯、熔断四项纪律都能在 atomic step 粒度生效。

### X2 不确定性路由

**[Must] 不变式**：执行中（任何时序条目或横切条目内）遇到**Agent 自身发现**的无法自主解决的不确定性时，必须立即按本条分级判据归类、按分流策略处理。禁止"先继续看看"、禁止合并处理多个不确定性、禁止跳过分级直接假设。

> X2 处理的是 **Agent 主动发现** 的不确定性。外部指出的问题（Agent 未察觉但被他人发现）走 X4 外部反馈处置。两者互补不重叠。

**分级判据**：

| 类别 | 客观判据 | 典型场景 |
|------|---------|---------|
| 阻塞性 | Agent 基于当前认知继续推进会产生错误结果 / 方向偏差 / 必须等待外部输入才能继续的状态 | 输入矛盾、权限未定义、核心业务规则缺失、无法唯一推断的关键假设 |
| 非阻塞性 | Agent 可用合理假设继续推进，且"假设与最终答案不一致"不会导致整体产出不可用（仅需回归性小修） | 格式边界、命名偏好、默认值选择、可事后调整的细节 |

**判据核心**：问"我现在做假设继续推进，产物是否仍然对下游可用？"—— 是 → 非阻塞；否 → 阻塞。

**分流策略**：

**阻塞性路径**：
1. 立即暂停当前条目的推进
2. 按通用上报模板整理
3. 通过角色参数矩阵定义的上报路径发送
4. 等待外部解答；**未获解答前不得恢复推进**
5. 解答到达 → 按解答更新认知（可能需要回溯 T2/T3/T4）→ 继续

**非阻塞性路径**：
1. **不暂停**，继续推进
2. 基于合理假设完成当前步骤，假设写入进度文件「当前处理方式」字段
3. 按通用记录模板挂账到非阻塞清单
4. T6 整体自检时确认所有非阻塞项已列入成果文件问题清单
5. 交付时随产物附清单，由下游节点 / 反馈方确认

**通用模板骨架**：

```
【{阻塞 / 非阻塞} - {角色}】
所属条目：T[N] / X[N] / atomic step
编号：{前缀}-{序号}              # 前缀由参数声明文件定义
描述：[具体描述不确定性内容]
影响范围：[涉及的条目 / 步骤 / 下游节点]
{类型特有字段}：                  # 阻塞：待确认内容
                                  # 非阻塞：当前处理方式 + 待确认内容
{角色特有字段}：                  # 如某角色独有的"问题来源"等附加字段
```

角色在参数声明文件中声明各字段具体值，但**模板骨架与字段语义不得更改**。

**参数化维度**：
- 阻塞 / 非阻塞编号前缀
- 上报对象与路径
- 记录写入位置
- 角色特有字段

**不变式原因**：未分级 / 误分级的三种典型失败——①阻塞当非阻塞继续 → 沿错误方向 rework 整段；②非阻塞当阻塞暂停 → 无谓拖慢；③不分级"先继续看看" → 累积不确定性形成认知断层。二分判据 + 强分流路径同时拦截三种失败。

### X3 逐步自检

**[Must] 不变式**：每一个 atomic step 完成时，必须立即对照该步的**预设验收点**做验证。验证未通过则当步作废、回到起点重做；通过后才可标 `[x]` 并进入下一步。禁止"先把所有步骤做完再统一检查"。

**验收点来源**：T4 深入分析产出的 atomic step 清单，每个 step 必须附带验收点，可表达为：
- Given-When-Then 用例
- 具体 checklist 条目
- 对应规范文件的具体条目引用

**失败处理**：
- 轻度不符（格式 / 笔误 / 小偏离）→ 当步内修正后重新验证
- 中度不符（step 内部逻辑问题）→ 整步作废重做
- 重度不符（step 设计有误，说明上游条目拆解有问题）→ 走回溯总则，回到产生该 step 设计的上游条目：
  - T1 atomic step 失败（如派发清单文件无法读取）→ 走 X2 阻塞向派发方澄清
  - T2 / T3 / T4 atomic step 失败 → 回到该条目内重新拆解（必要时回前一时序条目）
  - T5 atomic step 失败 → 回 T4 重拆
  - T6 atomic step 失败 → 详见 T6「失败处理」（按问题属性映射）

**工具核查要求**：
- 凡可用工具验证的条目（编号唯一性、文件存在性、关键字计数、格式合规性等），必须用工具验证，禁止肉眼核对
- 工具核查命令必须在进度文件的 step 验收点中明确写出（具体命令格式由参数声明文件按角色业务定义，方法论不规定）

**参数化维度**：
- 单步验收点的格式约定
- 失败时的处理粒度
- 工具核查的具体命令集

**不变式原因**：错误在产生的时刻修复成本最低。等到下游 step 基于错误继续后，修复要同时撤销 N 个下游工作。X3 与 X1 共生——X1 管记录、X3 管验证，共同构成"真正完成一步"的完整语义。

### X4 外部反馈处置

**[Must] 不变式**：Agent 在执行中或交付后收到**外部来源**指出的问题或反馈时，必须按本条处置流程处理。禁止忽略、禁止 partial 修复、禁止在未完成反馈处置前声明交付完成。

> X4 处理的是 **外部发现、Agent 未察觉** 的问题（即"盲区补偿"）。Agent 自己发现的不确定性走 X2。

**外部来源的通用类别**（具体到本工作流的反馈渠道见参数声明文件 X4 反馈来源入口）：

- **监督角色反馈**：上游或同级监督 Agent（如审核者）指出的问题或整改意见
- **需求方反馈**：任务发起方 / 决策方在执行中或交付后提出的调整意见（含临时插入的修改要求）
- **变更请求**：工作流变更管理机制传入的需求 / 范围 / 边界变更
- **工具链信号**：自动化校验工具、CI、脚本返回的错误或警告信号
- **下游 Agent 反馈**：下游消费方在使用上游产出时发现的问题

**反馈接收格式要求**：

外部反馈到达时应包含三要素（缺失则走 X2 阻塞向反馈方澄清）：
1. 问题描述
2. 涉及范围
3. 反馈性质（必须整改 / 建议调整 / 启发思考）

**处置流程**：

1. **接收与理解**：完整读取反馈原文，禁止只看摘要。不理解时走 X2 阻塞向反馈方澄清，不得自行猜测反馈意图
2. **范围确认**：把反馈映射到方法论的具体条目（是 T2 初步理解错了？T3 遗漏 unknown？T4 计划偏差？T5 执行偏离？T6 整体有问题？），理清反馈的影响面
3. **处置路径选择**：

   | 反馈性质 | 处置路径 |
   |---------|---------|
   | 问题在当前条目范围内（轻度） | 当步修复后重新走 X3 验证 |
   | 问题在前序条目（中度） | 走回溯总则，回到对应条目更新 |
   | 问题跨多条目（重度） | 按最早涉及的条目回溯，级联更新下游 |
   | 问题触及根本方向 | 升级为阻塞，走 X2 路径由 Agent 反向上报"反馈本身需裁决" |

4. **处置执行**：按方法论正常流程，回溯路径涉及的条目重新执行，X1/X3 纪律照常保持
5. **反馈闭环**：处置完成后在进度文件标记反馈已消化，必要时向反馈方确认，禁止"自己觉得改完就算"
6. **`[Must]` 同位置/同现象反馈 ≥ 2 次 → 强制溯根因层**（2026-06-02 治本）：接同位置（同 file:line / 同章节 / 同视觉元素 / 同 PM 错诊位置）反馈 ≥ 2 次（即第 2 次反馈"X 仍未解决"/"X 还是太小"/"X 还是错的"等同型现象），**禁止继续治标**（仅修当前现象），**必须**派 Explore 调研 → 找上游真根因（如库行为 / 多层调用链 / 设计盲区 / 同型机制污染）→ 治本后再回告。**违反信号**：mermaid 全屏预览"默认比例小"踩了 3 轮反馈（issues #3/#6/#8 才溯到 mermaid useMaxWidth + getBBox 根源）；3 P0 critical 第 1 轮编排器误判后第 2 轮 PM 复述同位置反馈，编排器仍未识别该信号（issues #41/#42，已 commit `e043606` 元错型治本）。

**参数化维度**：
- 反馈来源及入口（按角色列出各自关心的反馈渠道）
- 反馈落盘位置
- 反馈处置完成的标记方式

**不变式原因**：

Agent 的自我检测存在根本盲区——T3 / T6 / X3 只能**降低盲区面积**，不能**消除盲区**。工作流的多角色协作设计本身就是为了让不同视角互为盲区补偿。方法论若不显式承认外部反馈的必然性并规定处置义务，会出现两种典型失败：

- **选择性响应**：修复显而易见的点，忽略深层问题（"反馈说 A 错了"→只改 A，不查 A 的根因是否影响 B/C）
- **就地修补**：在当前层面打补丁而不回溯根因（T5 执行中遇到反馈，直接改当前步骤输出，不更新 T4 的计划 → 同类错误会在后续步骤再次出现）

X4 的处置流程通过"范围确认 → 回溯到根因条目 → 级联更新下游"三步，强制 Agent 从根因层面消化反馈。

## 四、元规则

### [Must] 回溯总则

任何时序条目或横切条目在执行中发现**前序条目的认知已失效**（T2 漏的 unknown、T3 错的假设、T4 走偏的计划、T5 执行中暴露的新依赖）时，**必须回溯到对应条目更新后再继续**，禁止"就地修补"、禁止"假装前序已完成"。

**回溯本身不违反方法论；沉默地跳过回溯才违反。**

### [Must] 迭代熔断

每个时序条目作为「被回溯目标」累计被回溯不得超过 **3 轮**——T4 被回溯（无论从 T5 / T6 / T3↔T4 收敛兜底 / X4 反馈 / X3 重度失败 等任何源触发）累计 3 轮即触发熔断；T3 / T2 被回溯各自独立计数同样累计 3 轮触发熔断。超过即升级为元问题（"我无法判断当前状态是否可以继续"）走 X2 阻塞上报，由外部裁决：继续迭代 / 放弃本次任务 / 改变任务边界。

**「一轮回溯」的定义**：一次完整的"触发回溯 → 回到目标条目执行 → 回到原触发点"循环算作一轮，无论中间撤回了多少 atomic step。计数粒度是**条目级**而非 atomic step 级——这是熔断阈值定为 3 的前提。

**计数粒度**：以"被回溯目标条目"为单位独立计数。例如 T4 被回溯 3 轮触发自身熔断，但 T3 被回溯 1 轮不会被 T4 熔断牵连。各条目计数独立。

**触发源涵盖**：所有回溯触发源均纳入对应目标条目的同一计数——
- Agent 自主迭代（如 T3↔T4 收敛失败、T5 偏离回溯、T6 失败处理触发的回溯）
- X3 重度失败触发的回溯
- X4 外部反馈触发的回溯（监督反馈 / 需求方反馈 / 变更请求 / 工具链信号 / 下游反馈等）

不区分触发源——避免 X4 反馈成为绕过熔断的渠道。

熔断对所有回溯路径生效（T4 ← T5/T6/T3 等、T3 ← T4/T5 等、T2 ← T3/T4/T5 等），不仅限于 T3↔T4。

## 五、参数声明文件的契约

本方法论的**参数槽位**由各 T/X 条目的"参数化维度"段定义。**实际参数值**由配套的参数声明文件填入，不在本方法论文档内。

### 5.1 参数声明文件的位置与命名

参数声明文件由各工作流自行命名与维护。本工作流当前使用 `pm-workflow/rules/agent_parameters.md`。

### 5.2 参数声明文件必含内容

任何符合本方法论的参数声明文件，至少包含以下两节：

| 节 | 内容 |
|----|------|
| **主参数矩阵** | 行 = 本方法论各 T/X 条目「参数化维度」槽位；列 = 各 Agent；格 = 该角色在该槽位的具体值 |
| **角色 atomic step 声明表** | 按角色 → T1-T6 顺序展开；每个 T 条目下列出该角色的 atomic step 切分清单 + 验收点 |

### 5.3 与本方法论文档的对应关系

- 本方法论文档**只声明槽位**——读者从各 T/X 条目的"参数化维度"段了解需要哪些参数
- 参数声明文件**填入具体值**——读者从其中了解某具体角色在本工作流的实例化
- 两者一对一对应，方法论新增槽位 → 参数声明文件须同步增行
- subagent 派发时同时传入两份文件

### 5.4 新增 Agent 类型的接入流程

详见参数声明文件本身的「新增角色指南」。原则：

- 不得修改本方法论的任何不变式条目
- 在参数声明文件的主参数矩阵增加一列，对照槽位填值
- 在角色 atomic step 声明表增加该角色的 atomic step 清单（须通过 X1 配套四条判据自检）

## 六、SSOT 主从架构与表述精确性原则

> **本节定位**：定义工作流文档体系的**单一事实源（SSOT）主从架构**与**表述精确性**两项跨文件原则——所有 Agent 规范、模板、命令文件、兜底脚本须遵守。本节是对历史复盘识别的「概念重复表述、主从倒置、表述歧义」根因的方法论级回应。

### 6.1 SSOT 主从架构原则

`[Must]` 任何在多个文档中出现的核心概念（如 owner 归属、atomic step 切分、子产物状态、模块进度并发、组件 D 维度反查、整改循环计数等），须显式声明四要素：

| 要素 | 内容 |
|------|------|
| **唯一权威源（SSOT）** | 哪个文件 + 字段路径是真源（如 `scaffold.modules[].owner_assignments`） |
| **派生视图清单** | 列出所有派生 / 复刻 / 反向引用的下游文件（如"任务卡 C 表 / PRD 草稿 OWNER-INFO 注释 / components A 表"）|
| **调整方向** | 先改源、再同步派生；**禁止反向**（禁止派生改了主源不改） |
| **双向引用** | 源 → 派生 / 派生 → 源 必须有显式 link，跨文档时尤其严格 |

`[Must]` SSOT 主源声明须放在主源文件的对应章节顶部（如 schema 段、矩阵段），派生视图须在自身章节内显式 link 到主源。

`[Should]` 同一概念在 ≥ 3 个文档出现时，应在主源文件追加"流转链路图"（计算 → 存储 → 读取 → 校验），作为单一权威表述供所有派生视图引用。

`[Recommended]` 实际成果文件（如各阶段成果、components_*.md、scaffold.json）的字段也适用 SSOT 原则——下游不得就地纠偏，须回溯到 SSOT 源修正后再执行下游。

### 6.2 表述精确性原则

`[Recommended]` 涉及编号体系 / 字段路径 / 执行主体 / 时机的表述，建议采用**机械语言**（精确路径、阶段编号、执行主体）替代纯自然语言：

| 类型 | 自然语言（歧义）| 机械语言（精确）|
|------|---------------|---------------|
| 字段引用 | "owner_assignments 字段" | `scaffold.modules[].owner_assignments`（含路径深度） |
| 阶段编号 | "任务规划阶段" | `Step 1（任务规划，PM）`（含执行主体） |
| 时机表述 | "Spec Agent 编写时" | `Step 1 范围预分配 + Step 3 顺序填入`（含两段式） |
| 执行主体 | "由系统/工具" | "由编排器（不读角色规范全文）/ 由 PM Agent / 由 gen_scaffold.py 脚本"（明示主体）|
| 状态/动作 | "预算" / "处理" / "执行" | "从 X 字段读出后注入 Y" / "按 N 维度逐项判定" / "调用 X 脚本"（明示具体动作）|

`[Should]` 中文表述涉及核心概念时，应在括号内辅以精确机械路径（如"配置项（`scaffold.modules[].owner_assignments`）"）。

`[Should]` 表述歧义触发的多义解读（如"分配时机"既可指范围预分配也可指具体编号填入）须在表述本身用括号穷尽；不得依赖"上下文应该能看懂"。

## 七、新规则建立的边界思考清单

`[Should]` 新建规则/规范条款时，思考三类边界：

### 7.1 例外场景

是否存在合理的"打破规则"场景？需声明局部豁免吗？

- 例：mermaid 全禁规则在用户旅程场景需局部豁免 → 应在初版规则中显式声明豁免清单
- 例：interaction-card 不嵌套规则在某些极端场景（如多帧并列对比）是否需例外？
- 检查：列出至少 3 个"违反本规则但合理"的潜在场景，判断是否需在规则中预留豁免

### 7.2 同族对称性

是否引入了不对称结构？同族成员是否需同步补全？

- 例：token 系统加 `--fb-error-bg-light` 时，是否也需 `--fb-success-bg-light` / `--fb-warning-bg-light`？
- 例：阶段 4 有 precheck 时，阶段 1/2/3 是否也需对称建立？
- 检查：识别本规则所属"族"，列出族内其他成员，判断是否同步补全或显式说明"暂不需要"

### 7.3 演化兼容

未来扩展时本规则是否会成为障碍？是否需预留扩展点？

- 例：scaffold.json schema v1 字符串数组 → v2 升级为对象数组时引入硬切换；初版若考虑可演化设计可降低升级成本
- 例：SSOT 双锚清单 A/B/C 三组分组是否能容纳未来新维度（如"机械化但仍存在边缘漏检"）？
- 检查：思考 N=2 / N=3 / N=10 倍当前数据规模时，本规则是否仍适用？是否需预留扩展点？

**应用范围**：所有标 `[Must]` 的新规则建立前必走本清单；标 `[Should]` 的新规则建议走（不强制）。

## 八、与其他规范文件的关系

本方法论**不重复**以下类别文件的内容，仅约束 Agent 的"如何做"层面：

| 文件类别 | 职责 | 与方法论的关系 |
|---------|------|--------------|
| **硬规则规范类** | 各阶段具体产物必须满足的"做什么"约束 | 方法论的 X3 / T6 验收点引用此类规范条目 |
| **产物格式规范类** | 产出物的结构、字段、表达样式规范 | 角色 atomic step 的验收点常以此类规范条目作为对照基准 |
| **内容模板类** | 各阶段成果的内容模板与填写示例 | 角色 T5 主体执行的产出格式锚点 |
| **参数声明文件** | 本方法论的实例化伙伴文件 | 与本方法论一对一对应，subagent 派发时同步传入 |
| **角色主文档** | 各 Agent 的角色规范、阶段任务、自审清单等角色特有内容 | 头部声明遵循本方法论 + 链接对应参数声明文件；不得重复定义参数 |

Agent 执行时须**同时遵守**本方法论和上述文件——**方法论管流程正确性，其他文件管产物正确性**。

本工作流当前的具体规范文件清单见各 Agent 主文档的「依赖规范声明」段或参数声明文件。**本方法论文档不绑定具体文件名**——理论上可剥离当前工作流被其他工作流复用。

**与角色主文档的边界**：本方法论是所有 Agent 的**共享底座**，各 Agent 主文档在此之上只做**参数声明指针 + 角色特有动作定义**，不得与方法论条目产生冲突或重复定义。
