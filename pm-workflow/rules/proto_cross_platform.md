## 一、多端覆盖原则

### `[Must]` 平台覆盖要求

原型须严格按产品定义「目标平台」字段确定覆盖范围：

| 产品定义目标平台 | 原型覆盖要求 |
|------------|------------|
| 仅 APP 端 | 所有页面绘制手机帧和 PAD 帧 |
| 仅桌面端 | 所有页面只绘制桌面帧 |
| 双端（APP 端为主） | 所有页面先绘 APP 帧（手机+PAD），再绘桌面帧；布局完全一致的可标注「布局同 APP 端」并简化 |
| 双端（桌面端为主） | 所有页面先绘桌面帧，再绘 APP 帧；布局一致的可简化 |
| 双端（权重相同） | 所有页面完整绘制两端帧，不得简化任一端 |

**「布局一致」判断标准**：两端页面结构、组件层级、操作入口完全相同，仅尺寸不同。任何组件位置或操作路径差异，均视为需完整绘制。

## 二、骨架屏引用规范（跨端同构页面简化）

### 适用场景

当同一页面在不同端口功能完全相同、仅布局结构因尺寸不同而缩放时，允许使用骨架屏引用替代完整绘制该端帧。

### `[Must]` 触发条件（同时满足所有条件才允许使用）

1. 两端字段集合完全一致（无字段增减）
2. 组件类型和排列层级完全相同（无操作路径差异）
3. 交互触发方式相同（无移动端特有手势、无桌面端特有 hover 态）
4. 导航结构相同（无底部 Tab 与侧边栏的切换差异）

### `[Must]` 骨架屏帧最低要求

骨架屏不是纯文字说明卡，必须是具有视觉结构的帧，须包含：

| 要求项 | 规则 |
|-------|------|
| 功能分区 | 导航区、内容区、底部操作区须通过背景色差异（`var(--fb-bg-5)` 占位块）或分隔线区分 |
| 按钮占位 | 一个按钮对应一个独立占位块，数量与参考帧完全一致 |
| 关键入口 | 核心操作按钮必须有占位，不得因骨架化而省略 |
| 标注引用 | 帧标题行必须标注参考帧 ID，格式：「布局同 [P-03 手机帧]，帧尺寸按平台规范替换」 |

### `[Must]` 禁止使用骨架屏引用的情形

以下情形必须完整绘制该端帧：

- 两端有不同的交互触发方式（如移动端下拉刷新 vs 桌面端分页控件）
- 两端导航结构不同（如手机底部 Tab vs 桌面端侧边栏）
- 两端有角色权限差异
- 两端存在端口特有的功能入口（如 APP 端扫码、桌面端键盘快捷键）

## 三、移动端操作布局规范（v2 按页面状态分类）

> 适用端口：APP（手机 + PAD）、H5、小程序
> 核心目的：按业务场景分类，规范每类场景的顶部 navbar variant + 底部 action-bar 合规性。
>
> **v1 → v2 迁移说明**：v1「按操作数量分类」（1/2/≥3）被实证证明过严且歧义大（详情页主操作 4 并列实际合理）。v2 改为「按页面状态分类」（7 场景），每场景明示 navbar variant + 底栏使用规则。precheck 按 PM 自报 `data-variant` / `data-purpose` 字面判定，零启发式，避免 v1 启发式 5/5 FP 教训。

### `[Must]` 7 场景 navbar variant + 底栏使用矩阵

| 页面状态 | data-variant | navbar slot 填充 | 底栏 .fb-action-bar 合规 data-purpose |
|---|---|---|---|
| **次级详情页**（浏览态）| `detail` （默认）| `.fb-nav-back`+`.fb-nav-title`+`.fb-nav-action` ≤1 | `page-main`（主操作 ≤4 并列）|
| **列表多选态** | `multi-select` | `.fb-btn-text`（取消）+`.fb-nav-title`（已选 N/M）+`.fb-btn-text`（全选/反选）| `multi-select-batch`（批量操作 ≤3）|
| **表单/浮层确认** | `confirm` | `.fb-btn-text`（取消）+`.fb-nav-title`+`.fb-btn-primary`（确认）| 通常无底栏 |
| **编辑态密集操作** | `edit` | `.fb-btn-text`（完成）+`.fb-nav-title`+`.fb-nav-action`（···）| 通常无底栏 |
| **列表入口页** | `list` | `.fb-nav-back`（可选）+`.fb-nav-title`+`.fb-nav-action` × N（筛选/搜索/新增）| 通常无底栏 |
| **向导分步** | `workflow` | `.fb-nav-back`（上一步）+`.fb-nav-title`（步骤 X/N）+`.fb-nav-action`（跳过）| `workflow-nav`（上/下一步）|
| **主页 / 一级 Tab** | `home` | —+`.fb-nav-title`（logo）+`.fb-nav-action` × N（搜索/我的）| `fb-tab-bar`（非 .fb-action-bar）|

### `[Must]` data-variant 显式声明

PM 写 `.fb-navbar` 时**必须**加 `data-variant="XXX"` 属性明示业务场景。precheck 按字面判定（零启发式 + FP ≈ 0）。default `data-variant="detail"`（向后兼容现有用法，未声明等同 detail）。

详 navbar HTML 模板见 `bujue-design-system/fb-fallback-manifest.md §3.12 v2`。

### `[Must]` 底栏 data-purpose 合规白名单（移动 frame 直接子层 .fb-action-bar 例外）

`.fb-action-bar` 在 `.phone-frame` / `.h5-frame` / `.miniprogram-frame` 直接子层默认 WARN（S4-28 v3 R2），**例外**：含合规 `data-purpose` 属性的场景：

| data-purpose | 合规场景 | 关联 navbar variant |
|---|---|---|
| `page-main` | 详情页主操作（≤ 4 个并列）| `detail` / `edit` |
| `multi-select-batch` | 多选态批量操作（≤ 3 个并列）| `multi-select` |
| `workflow-nav` | 向导上一步/下一步 | `workflow` |

其他场景（无 `data-purpose` 或非白名单值）仍 WARN（应走 navbar variant + 「···」菜单 或 改 modal/drawer 内）。

### `[Should]` 操作数量 vs 优先级（替代 v1 收纳阈值）

按 v2 框架，操作数量本身**不再是分类轴**，但仍有"信息密度"参考：

| navbar variant | 单 slot 操作数推荐 | 超出怎么办 |
|---|---|---|
| `detail` right slot | ≤ 1（主操作）| 收纳到 `.fb-nav-action`「···」菜单 |
| `multi-select` left/right slot | 1（取消 / 全选）| 不应超出 |
| `confirm` left/right slot | 1（取消 / 确认）| 不应超出 |
| `edit` right slot | ≤ 2（···菜单 + 1 主操作）| 收纳菜单 |
| `list` right slot | ≤ 3（图标按钮组）| 收纳到「···」菜单 |
| `workflow` right slot | ≤ 1（跳过）| 不应超出 |
| `home` right slot | ≤ 3（图标按钮组）| 收纳菜单 |

### `[Must]` 「···」菜单规范

- 点击「···」→ 底部抽屉弹出，列出收纳的操作清单
- 清单排序：常用操作在上，危险操作（删除、清空等）在底部，以红色文字区分
- 危险操作触点用 `D[ZZ]` 编号（非 `T[ZZ]`），spec 触点表对应 D 类
- 若页面无任何次级操作，可省略「···」按钮

quotation-tool prd 5 处 phone-frame 直接子层 .fb-action-bar 在 v2 框架下转合规路径：

### `[Must]` 反 pattern（v1 自由发挥旧坑）

| ❌ 错误做法（v1 实证）| ✅ v2 正确做法 |
|---|---|
| 多选态顶部用 `<div style="position:sticky;top:0">` inline 自由发挥 | 用 `.fb-navbar data-variant="multi-select"` 标准 slot 填充 |
| 详情页 4 个并列按钮直接放底部 `.fb-action-bar`（无 data-purpose）| 加 `data-purpose="page-main"` 明示合规场景 |
| 表单页底部放「保存」按钮（无 navbar 确认 variant）| 用 `.fb-navbar data-variant="confirm"` 右 slot 放 `.fb-btn-primary`（确认）|
| 自由发挥自定义 class（`.toolbar` / `.top-bar` / `.action-strip` 等）| 必走 `.fb-navbar` + variant；precheck `check_panel_class_evasion` 扩展拦截 |

### `[Should]` 卡片/列表项操作规范

「卡片操作」指对列表中单条数据项的操作（如「编辑」「删除」「复制」等）。

**基本原则**：卡片/列表项的行内常驻操作按钮原则上不超过 1 个；超过 1 个时，通过**长按触发上下文菜单**统一收纳。

**允许直接在卡片上展示的例外**（仅以下两种）：

| 场景 | 理由 |
|------|------|
| 步进器（数量 +/-，如购物车） | 数量调整属实时频繁操作，需直接响应 |
| 列表项右侧「›」导航箭头 | 表达导航语义，不属于行为操作按钮 |

**长按菜单规范**：
- 用户长按卡片/列表项 → 底部抽屉或气泡菜单弹出，列出可用操作
- 菜单排序：普通操作在上，危险操作（删除）在底部，以红色文字区分
- 需绘制「[叠加态-长按菜单]」帧，展示菜单弹出后的视觉状态

**批量操作入口**：通过长按列表项弹出上下文菜单，菜单中提供「多选」选项，点击后进入批量选择模式（每行出现复选框，导航栏切换为「取消 / 批量操作」工具栏）。

### PM Agent 执行时的判断流程

PM Agent 在设计每个页面时，按以下流程判断操作布局：

```
Step 1：列出该页面所有操作
  ↓
Step 2：判断页面级操作数量
  · 1-2 个 → 直接展示，确定主次按钮
  · ≥3 个  → 选最高频 1 个直接展示，其余收入「···」菜单
  ↓
Step 3：判断卡片操作类型
  · 步进器/单一核心操作 → 直接在卡片上展示
  · 其他操作（编辑/删除/分享等）且数量 ≥ 2 → 收入长按菜单
  ↓
Step 4：需绘制的叠加态帧
  · 有「···」菜单 → 绘制「[叠加态-更多菜单]」帧
  · 有长按操作   → 绘制「[叠加态-长按菜单]」帧
  · 有批量选择   → 绘制「[叠加态-批量选择]」帧
```

## 四、弹框 & 底部浮框统一交互规范

> 适用端口：APP（iOS / Android）、H5、小程序
> PM Agent 在设计任何弹框/浮框交互时，须先查阅本节决策树，再确定组件类型。

### 统一定义

| 组件 | 定义 |
|------|------|
| **中间弹框（Modal）** | 居中弹出，全局遮罩，强阻断，必须用户主动操作才能关闭 |
| **底部浮框（Bottom Sheet）** | 从底部滑出，半遮罩 / 全局遮罩，轻阻断，支持手势关闭 |
| ↳ 菜单型 | 纯选项列表：标题（可选）+ 列表选项 + 取消按钮 |
| ↳ 表单型 | 含输入框、选择器等复杂交互：标题栏（含关闭/取消 + 确定按钮）+ 表单内容区 |
| ↳ 面板型 | 持续操作面板（如礼物面板），可无蒙层 |

### 决策树（选对组件）

```
需要用户响应？
├─ 否 → Toast / 顶部轻提示
└─ 是 → 操作不可逆 / 涉及资金 / 强授权 / 强警告？
        ├─ 是 → 中间弹框
        └─ 否 → 需要强确认？（即使选项 ≤2 也需用户明确表态）
                ├─ 是 → 中间弹框
                └─ 否 → 选项 ≥3 / 多选菜单 / 上下文操作 / 复杂表单？
                        ├─ 是 → 底部浮框
                        └─ 否 → 中间弹框（1～2 个选项的简单确认）
```

### 使用规则

#### 必须使用「中间弹框」的场景

满足任意一条即使用：

- `[Must]` **操作不可逆、高风险**：删除数据、清空缓存、退出登录、解绑账号、取消订单、放弃已编辑内容
- `[Must]` **涉及资金 / 安全**：支付确认、充值、提现、实名认证、修改密码
- `[Must]` **系统级强制授权**：通知权限、相机 / 相册 / 定位 / 存储权限
- `[Must]` **必须用户明确同意**：隐私政策、用户协议、重要规则告知
- `[Must]` **强警告 / 异常提示**：账号异常、异地登录、强制更新、网络异常阻断操作
- `[Must]` **版本更新提醒**：强制更新、重要更新推荐
- `[Should]` **操作只有 1～2 个明确选择且需要强确认**

#### 必须使用「底部浮框」的场景

满足任意一条即使用：

- `[Must]` **操作是多选一菜单**：分享、举报、复制链接、打开方式、切换账号
- `[Must]` **当前页面上下文操作**：选择分类、筛选条件、选择规格 / 地址 / 时间
- `[Should]` **登录方式选择**：微信 / QQ / 手机 / 邮箱登录
- `[Should]` **选项数量 ≥3 个**，或需要可滚动列表
- `[Should]` **复杂表单类操作**：地址编辑、日期区间选择、筛选面板
- `[Should]` **非强风险、非强制确认的轻量选择类操作**
- `[Recommended]` 希望单手操作友好，不希望过度打断用户流程

#### 两者都不建议用（统一用轻提示）

- 普通操作结果：提交成功、删除成功、复制成功
- 无需用户确认的弱提醒

→ `[Should]` 统一使用 Toast / 顶部轻提示（自动消失，不阻断）

### 视觉与交互规范

#### 通用 DOM / z-index 层级约束（所有弹框 & 底部浮框适用）

- `[Must]` **z-index 上限 199**：弹框 / 浮框 / 遮罩相关任意 class（无论 PM 自定义 `.modal` / `.popup` / `.dialog` / `.bottom-sheet` / `.overlay` 还是 inline `style="z-index: ..."`）的 z-index 必须 **< 200**——`.section-header` 是 sticky 吸顶元素，z-index 锁定 200（`prd_template.html` 真源），弹框层级必须低于它，确保 sticky 标题在弹框打开时仍可见
- `[Must]` **`.frame-card` 禁止 z-index inline style**：frame-card 容器本身保持文档流默认 `z-index: auto`，**禁止**通过 `style="z-index: 999"` 或自定义 class 强制提升层级——这会盖住 sticky section-header
- `[Should]` **中保真原则**：弹框视觉帧应作为模块帧内的内容元素正常文档流定位（PRD 中保真原型不需要 fixed 模拟真实弹出效果）；如确需视觉示意 fixed/absolute 定位，须在父容器加 `position: relative` 隔离层叠上下文 + z-index < 200
- **SSOT 真源**：`prd_template.html` `.section-header { z-index: 200 }` + `prd_expression_standard.md §四 区块 A` z-index 层级保护规则；本节为多端弹框规范的 z-index 入口

#### 中间弹框

- `[Must]` 居中展示，圆角统一，带全局半透明遮罩
- `[Must]` 结构：标题 + 描述文案 + 操作按钮
- 按钮规则：
  - `[Recommended]` 默认 2 个按钮
  - `[Should]` 特殊场景允许 3 个按钮，但必须**纵向排列**
  - `[Must]` 严禁 4 个及以上按钮
  - `[Must]` 主按钮：确认 / 同意 / 继续（高亮强调）；次按钮：取消 / 放弃（弱化）
- 关闭规则：
  - `[Must]` **高风险场景**（支付、删除、退出、授权）：仅允许点击按钮关闭，禁止蒙层关闭
  - `[Should]` **普通信息确认**（活动告知、版本说明）：允许蒙层关闭
- 文案规范：
  - `[Must]` 按钮文案用动词，例如「删除」而不是「确定」
  - `[Should]` 标题不超过 1 行，描述不超过 3 行
  - `[Must]` 高风险操作必须说明后果，例如「删除后无法恢复」

#### 底部浮框

- `[Must]` 从屏幕底部平滑滑入，支持下滑手势关闭
- 蒙层规则：
  - `[Must]` 选择 / 确认类：必须有蒙层，支持点击蒙层关闭
  - `[Optional]` 持续操作类面板型：可无蒙层（白名单：音乐播放控制面板、地图 POI 详情）
- 菜单型：`[Must]` 必须保留「取消」作为最后一项；`[Should]` 危险操作用红色文字突出
- 表单型：
  - `[Must]` 结构：标题栏（左侧关闭/取消 + 中间标题 + 右侧确定/完成）+ 表单内容区
  - `[Must]` 必须提供明确的关闭入口和确认入口
  - `[Should]` 表单有未保存修改时，点击关闭触发中间弹框二次确认
- 通用：
  - `[Must]` 高度不超过屏幕高度 85%；`[Should]` 内容过长时标题固定，内容区滚动
  - `[Must]` **包含输入框时**：升级为全屏或大半屏样式，键盘弹出时禁用下滑手势关闭

#### 加载中 / 不可关闭状态

适用场景：支付处理中、订单提交中、文件上传中

- `[Must]` 禁用所有按钮，禁止任何方式关闭（蒙层、返回键、手势）
- `[Must]` 必须有超时兜底，超时后自动转为可关闭的失败提示
- `[Must]` 显示明确加载状态（转圈 + 「处理中，请稍候...」）

**超时时间参考**：

| 场景类型 | 推荐超时 |
|---------|---------|
| 支付处理 / 订单提交 | 15s |
| 短信验证码发送 | 10s |
| 文件上传（< 10MB） | 30s |
| 文件上传（≥ 10MB） | 60s，并显示进度百分比 |
| 数据同步 / 导出 | 30s |
| 实名认证 / 人脸识别 | 20s |

### 双平台统一行为兜底

#### 关闭优先级

- 中间弹框（高风险）：`[Must]` 仅按钮关闭
- 中间弹框（普通）：`[Should]` 按钮关闭 > 蒙层关闭
- 底部浮框-菜单型：`[Should]` 按钮关闭 > 下滑关闭 > 蒙层关闭
- 底部浮框-表单型：`[Should]` 标题栏关闭/取消 > 蒙层关闭（有未保存内容时需二次确认）

#### Android 返回键 / 手势返回处理

- `[Must]` 中间弹框（高风险）：拦截返回键，等同点击「取消」按钮
- `[Should]` 中间弹框（普通）：返回键直接关闭
- `[Should]` 中间弹框（单按钮）：返回键等同点击该唯一按钮
- `[Should]` 底部浮框：返回键 = 下滑关闭
- `[Must]` 加载中状态：拦截返回键，不允许关闭

> H5 环境下无法拦截 Android 系统返回键，在触点说明中注明「Android 返回键行为由浏览器控制」。

#### 键盘弹出处理

- `[Must]` 弹框内有输入框时：整体自动上推，不遮挡输入区域
- `[Must]` 底部浮框含输入框时：禁用下滑手势关闭，避免误触丢失输入

### 嵌套与堆叠规则

- `[Must]` 同一时刻只允许出现一个中间弹框
- `[Should]` 底部浮框上不允许再弹底部浮框；以下场景可申请豁免：表单型底部浮框内含二级选择字段（如地图选择），允许叠加，但 `[Must]` 最大堆叠层数不超过 **2 层**，且须在设计评审中登记
- `[Should]` 底部浮框上可以弹中间弹框（如选择地址 → 确认删除地址）
- `[Must]` 弹框出现时，Toast 应显示在弹框之上，而非被遮挡
- `[Must]` 用户冷启动后多个弹框需排队展示，不能同时叠出
- `[Must]` 任何形式的弹框堆叠不得超过 2 层（含底部浮框 + 中间弹框的组合）

### 频次与防打扰

#### 系统主动推送类弹框（非用户操作触发）

- `[Must]` 有明确的触发条件和业务目的（不允许无条件弹出）
- `[Must]` 经过产品 / 运营评审，纳入弹框管理后台统一调度
- `[Should]` 用户当前无正在进行的关键操作（填写表单、支付流程中）时才弹出
- `[Should]` 同一弹框 24h 内不重复弹出超过 3 次（强制更新除外）；同一会话内系统推送类弹框总数不超过 2 个
- `[Must]` 营销类弹框必须有「不再提醒」选项
- `[Should]` 新手引导类弹框完成后不应重复出现

#### 用户操作触发的反馈弹框

不受上述频次限制，但仍需遵守：
- `[Must]` 与用户操作直接相关，不夹带无关信息
- `[Should]` 操作结果为成功 / 非关键信息时，优先使用 Toast 而非弹框

### `[Must]` 禁止行为（红线）

违反以下任意条目视为缺陷：

- 禁止用底部浮框做删除、支付、退出等高危确认
- 禁止用中间弹框做长列表、多选项菜单
- 禁止中间弹框出现 4 个及以上按钮
- 禁止菜单型底部浮框不提供「取消」按钮
- 禁止表单型底部浮框不提供关闭/取消入口
- 禁止在加载态时允许用户关闭弹框
- 禁止高风险中间弹框允许蒙层关闭
- 禁止弹框内嵌套跳转到新页面而不关闭弹框
- 禁止弹框堆叠超过 2 层
- 禁止加载状态无超时兜底（无限等待）

## 五、流程图与外部依赖页面处理

### 流程图中的平台区分

prd.html 流程图中，APP 端与桌面端的同名流程分行展示，各自独立成行，不混合排列。APP 端行包含手机帧，PAD 帧如有差异单独标注。

### `[Should]` 设计范围外页面（外部依赖页面）处理规则

**定义**：不在本次需求设计范围内、但与本次需求存在业务关联的页面（属于其他已有模块）。

**基本原则**：
- 本次需求的功能页面中，不设置指向外部依赖页面的常驻导航入口
- 在原型中单独开「外部依赖说明」章节，说明各外部页面的模块归属及与本次需求的业务关系

**原型呈现方式（按依赖性质分两类）**：

**类型一：外部页面上的操作是流程关键步骤**（影响本次需求的数据或状态）
→ 生成**简化版页面**，只包含支撑该步骤所需的最少元素和触点，帧内标注「[外部模块-简化版]」

**类型二：外部页面仅作为流程的来源或去向**（只需知道「从这里来」或「到这里去」）
→ 以**占位节点**表达（标注页面名称和「[外部模块]」的方框），不绘制页面内容

**流程图处理**：外部依赖页面凡是某条流程的必经节点，均须出现在对应流程图中。若与本次需求仅有业务关联但不出现在任何流程路径上，只在「外部依赖说明」章节描述，不进入流程图。

## 六、多端并列展示视觉区分准则

> **适用场景**：同一 frame-card 内并列展示 ≥ 2 个不同端口的视觉帧（如桌面 + 手机 / 手机 + 平板 / 全端并列）。
>
> **核心目标**：因部分端口尺寸接近（如手机 H5 vs 小程序、桌面 vs 大平板），人类难以一眼分辨；本节统一约束多端并列时的视觉区分手段。

### 6.1 `[Must]` 端口标签 + 视觉帧 DOM 嵌套（`.frame-cell` 父子模式）

详见 `prd_expression_standard.md §四 区块 B`「端口标签 + 视觉帧 DOM 嵌套」`[Must]` 规则——本节作为跨端规范的入口，列出关键约束：

- **触发条件**：1 个 frame-card 内并列 ≥ 2 个不同端口的视觉帧
- **嵌套结构**：每端用一个 `.frame-cell` 容器把「`.frame-platform-item` 标签 + 视觉帧」绑定为父子嵌套；`.frame-cell` 作为 `.frame-wrapper` 的直接子元素，多 cell 横向并排
- **禁止**：端口标签**不得**放在 phone-frame / desktop-frame / tablet-frame / miniprogram-frame / h5-frame 等页面帧内
- **标签文本枚举**：`桌面` / `手机` / `平板` / `小程序` / `H5`（与 `proto_contract.md §五` 平台标签文本对齐）
- **核心机制**：标签与帧在同 `.frame-cell` 内（DOM 父子绑定）+ `align-items: flex-start` + cell 宽度自适应——zoom 缩 frame 后 cell 宽度收缩 → 标签紧贴对应帧。**完全 CSS-only，无需 JS sync**（DOMContentLoaded reflow = 0）
- **CSS class**：`.frame-cell / .frame-platform-item / .frame-platform-tag`（真源在 `prd_template.html`）
- **机械兜底**：`precheck_stage4.py check_frame_platform_tag` 在 Step 6.5 校验「frame-card 内并列 ≥ 2 端口时每个视觉帧必须包在 `.frame-cell` 内 + cell 内 `.frame-platform-item` + `.frame-platform-tag` 齐全」
- **单端豁免**：仅 1 个端口的 frame-card 可省略 `.frame-cell` + `.frame-platform-item`，`.frame-wrapper` 内直接放视觉帧

### 6.2 `[Must]` 端口差异说明（跟随端口标签）

> **[Must] SSOT 派生关系**：本节是「业务示例 + 渲染语义」派生层。**PAD 布局规则真源 = `proto_platform_app.md §三`**（含 PAD 默认横向 / 同手机帧豁免触发条件 4 项 / 骨架屏质量标准）；HTML 渲染约束真源 = `prd_expression_standard.md` 段「frame-platform-note」。本节示例（如"布局同手机帧"）的业务依据须查真源，**禁止从本节示例反推 PAD 规则**——示例只展示如何在 PRD 中**写**，不展示**何时可写**。详见 `ssot_anchors.md #47`。

详见 `prd_expression_standard.md §四 区块 B`「端口差异说明（跟随端口标签）」`[Must]` 规则（v1.1 新增，issue 2026-05-07_1400 条 5 落地）：

- **位置**：`.frame-platform-item` 内、`.frame-platform-tag` 下方独立一行 `<p class="frame-platform-note">说明文字</p>`
- **禁止**：端口差异说明（如"桌面端：布局同手机帧，空态居中显示"、"小程序端：布局同 H5"）**不得**放在视觉帧内
- **建议长度**：≤ 30 字，描述本端口的差异要点
- **可选性**：单端产品 / 端口间无差异时可省略 `.frame-platform-note`
- **CSS class**：`.frame-platform-note`（真源在 `prd_template.html`）

**典型场景**：

| 主端口 | 副端口 | 副端口的 frame-platform-note 示例 |
|-------|-------|--------------------------------|
| 手机 | 桌面 | "布局同手机帧，空态居中显示" |
| 手机 | 平板 | "布局同手机帧，宽度 768px" |
| H5 | 小程序 | "布局同 H5，无返回按钮" |
| 桌面 | 手机 | "底部 Tab 替代侧边栏导航" |

### 6.3 `[Recommended]` 端口排序约定

多端并列时建议自左向右按"桌面端 → 平板端 → 手机端 → 小程序 → H5"顺序排列，对应 `desktop-frame → tablet-frame → phone-frame → miniprogram-frame → h5-frame`。统一排序便于阅读时建立空间记忆。

例外：当产品定义某端口为主时（如「APP 端为主」），可将该端口前置展示。

### 6.4 `[Recommended]` 端口尺寸缩放比

多端并列时 frame-type 默认尺寸（来自 `prd_template.html`）：

| 端口 | frame class | 默认宽度 |
|------|-------------|---------|
| 桌面 | `.desktop-frame` | 自适应（最大 1440px）|
| 平板 | `.tablet-frame` | 768px |
| 手机 | `.phone-frame` | 440px |
| 小程序 | `.miniprogram-frame` | 375px |
| H5 | `.h5-frame` | 375px |

**[Should]** 同 frame-card 内多端并列时，**不得**强制缩放任一端使其与其他端尺寸接近——视觉尺寸差异本身是端口区分手段，强制等高 / 等宽会消解这一区分信号。

### 6.5 `[Optional]` 端口配色 / 边框区分

frame-type 默认无端口区分配色（共用 `var(--fb-border-1)` 边框 + `var(--fb-white)` 背景）。**[Recommended]** 如设计需要进一步强化端口识别，可通过暗色模式自动反转或临时附加 modifier class（如 `.frame-mobile-tinted`）实现，但**不得**在不同 PRD 中使用不同的端口配色——端口配色一旦引入须通过 `fb-fallback-manifest.md` 登记为统一规则。

### 6.6 与 `proto_contract.md §五` 文档文本层平台标签的关系

| 层 | 标识形式 | 用途 |
|----|---------|------|
| **文档文本层**（`proto_contract.md §五`）| 章节标题用文字标注（如 `P-03 [手机]`）| 章节级区分，便于目录与跳转 |
| **视觉层**（本节 6.1 / 6.2）| frame-wrapper 内 `.frame-cell` 嵌套 `.frame-platform-tag` + 可选 `.frame-platform-note` 端口差异说明 | 帧内一眼分辨 + 端口差异要点跟随标签可见 |

**两层互补**：文档文本层负责章节归属，视觉层负责帧内辨识，**不可互相替代**。

> **本节落实依据**：`pm-workflow/rules/workflow_maintenance_protocol.md`「模板/规范描述粒度原则·文档层 ↔ 视觉层对偶」`[Must]` 原则——多端并列是文档层标识（端口名）↔ 视觉层标识（端口标签）对偶的典型场景，本节统一约束所有"多端并列"复合呈现需求，避免未来零散补漏。

## 七、`[Must]` 生产前判定标准前置（PM 阶段 4 开工必读）

> **为什么有本节（根因 C 防御）**：以下判定标准在历史下游产品中**反复试错才确立**——PM 边生产边摸索、或等产品总监第 N 轮反馈才定，造成多轮返工。本节把已确立的判定标准**前置固化**为 PM 阶段 4 开工前必读的决策规则：**生产前先按标准定，不靠直觉、不等事后纠**。每条只给"判定路径 + 默认选择"，详细规范见各指针文件（不复刻，避免双源漂移）。

### 7.1 `[Must]` 端适配 vs 组件拆分判定（PRD 层一个组件 + 端适配）

**判定问题**：同一业务功能要覆盖多端时，PRD 层是"一个组件 + 端适配"，还是拆成 N 个端专属组件 / 页面？

**默认标准**：**PRD 层一个组件（一个范式 archetype）+ 端适配**。区域增减优先回 archetype 容纳契约（平台无关）；**仅当 2D 排布 / 占比跨端实质发散**（如同范式 phone 竖叠+底导 vs desktop 侧栏+多列）才拆 per-platform 表达。

**判定路径**：
1. 跨端只是宽窄 / 间距差异 → 一个 agnostic 骨架，端适配交给 frame 类 + fb-fallback CSS（不拆组件）
2. 跨端区域有增减 → 优先回 #39 archetype 调容纳契约，不拆组件
3. 跨端 2D 排布 / 占比真不同 → 拆 ` ```skeleton:{platform} ` per-platform 块（一范式内 EITHER 1 个 agnostic OR ≥1 个 per-platform，不混用）

**详细规范**：SSOT 双锚 #41 / `rule_hard_constraints.md S4-32` + `proto_spec_md.md §四`「条件 per-platform」+ 本规范 §六（多端并列展示视觉区分）。

### 7.2 `[Must]` 次级页面返回入口判定（加 / 豁免）

**判定问题**：当前页面要不要加返回入口（移动端 `fb-navbar` 含 `fb-nav-back`）？

| 页面性质 | 返回入口 |
|---------|---------|
| 业务流子页（从某入口进入的次级页面：详情 / 编辑 / 表单 / 结果页等）| **加** `fb-navbar` 含 `fb-nav-back` |
| 底部 Tab 主导航直达页（用户经 Tab 切换而非"返回"抵达的主页面）| **豁免**（无 back，靠 Tab 导航） |
| H5 / 浏览器环境依赖原生返回的页面（浏览器后退键 / 手势已覆盖返回语义）| **豁免**（依浏览器原生返回域） |

**默认**：拿不准 → 业务流子页一律加；只有"主导航直达"与"浏览器原生返回域"两种情形才豁免。

**详细规范**：本规范 §三（导航栏布局 + `fb-navbar` 落地）+ `rule_hard_constraints.md S4-35` + SSOT 双锚 #45（机械兜底 `check_back_entry`，WARN 阶段，零启发式）。

### 7.3 `[Must]` 列表 / 数据展示选型判定

**判定问题**：数据用列表 / 表格 / 卡片 / 详情 / 海量虚拟列表？异常态（空 / 错 / 加载）怎么表达？

**默认标准**：按数据规模、字段数、操作密度走选型决策路径——**生产前先定选型，不边画边换**。

**详细规范**：`proto_data_display_selection.md`（列表 / 详情 / 异常态 / 海量数据选型决策路径，源 #18~22 确立）——该文件平时标"按需 Read 非必读"，但**阶段 4 含列表 / 表格 / 数据展示页时，升为生产前必读**。

## 八、端类差异化承载方法论（[Should]，2026-06-02 治"桌面端规约直接照搬移动端"反 pattern）

`[Should]` PM 写 spec 时必须为**每个端类（desktop / tablet / phone / mp / h5）单独评估**视觉 / 交互 / 操作复杂度，**禁止**桌面端规约直接照搬到移动端。判定时按以下优先级：

### 8.1 优先级 1：pub 库多端变体规范（pub 已含则严格沿用）

如 pub 库（`bujue-design-system`）对该组件已明示端类变体（按 Size 轴 / Platform 轴 / Density 轴），**直接沿用 pub 变体**（与 `fb-design-query/SKILL.md §二.X 决策树执行完毕后的显式声明` 一致 — 禁止 PM 凭直觉自由发挥构造端类差异）。

### 8.2 优先级 2：pub 库无明示时按通用约束判定（不穷举具体规则）

pub 库无对应端类变体时，PM 按以下 4 维度评估端类差异，并将判定逻辑显式记录到 spec：

| # | 维度 | 判定方向 |
|---|------|---------|
| 1 | **操作密度** | 屏幕物理尺寸 vs 操作目标数量；桌面 1440×900 vs phone 375×812 触达密度差异 ≈ 4-6 倍 |
| 2 | **触屏 vs 键鼠** | 点击区域最小尺寸（移动 ≥ 44pt / 桌面 ≥ 28px）+ 悬停态可用性（移动无 hover） |
| 3 | **单手 vs 双手操作** | 手机端可达区域（拇指 reach zone）+ 屏幕底部主操作 vs 顶部辅操作 |
| 4 | **端类业务承载差异** | 某些功能在某端不适用（如桌面侧栏在 phone 折叠为汉堡菜单 / pad 多列展示在 phone 单列堆叠）；业务侧 NB 决策 |

### 8.3 优先级 3：业务侧端类差异 NB 引用（spec 引 NB 真源不预设具体业务规则）

业务侧 NB 决策（如某产品的端口差异化承载 NB）作为本规范的下游消费案例 — **spec 引 NB-XXX 真源**，不在本规范预设具体业务规则。

### 8.4 反 pattern 警示

- ❌ spec 仅写"全端通用"或仅 desktop 规约而**未单独评估** phone-frame 适配 → 触发 PM 自审「跨平台差异化承载」条款验收失败（参 SSOT #61 §三.5 .1「跨平台关键差异」5 维分点 + 本节方法论）
- ❌ phone 端布局直接照搬桌面（如多列横排 / 桌面悬停态在移动端不可触达）→ 违反优先级 2 中触屏 / 单手维度
- ❌ 不查 pub 库多端变体直接自定义 → 违反优先级 1（先 pub 后自决）

### 8.5 本规范的边界（不规定具体方案）

本规范**不规定**具体规则（如按钮排列方式 / navbar 简化方案 / 底栏完整性 / hero 图占位高度等）— 此类具体规则属：
- **pub 库**（视觉真源，bujue-design-system）
- **业务侧 NB**（端口差异化承载特定决策）
- **spec 模块层**（模块特定 archetype.regions）

**本规范层级 = 方法论 + 优先级 + 反 pattern**，避免规则过于针对当前业务缺乏普适性。

- **本节 8 §端类差异化承载方法论** = L2 跨平台层级（PM 判定 spec 描述差异时的方法论）
