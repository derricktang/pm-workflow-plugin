# 数据展示方式选型规范（proto_data_display_selection）

> **文件性质**：`[Recommended]` 决策路径集合（引导 PM 思考），非硬约束。业务场景千变万化，边界 case 须 PM 判断 + 与产品总监确认。
>
> **适用阶段**：
> - 阶段 2 功能规划（页面信息架构选型时按需对照）
> - 阶段 4 PRD 实现（具体写法选型时按需对照）
>
> **触发方式**：PM 在每个页面设计时**按需 Read**。**不进**编排器派发 PM 的必读路径清单（避免每次派发都拉全文增加 token 成本）。
>
> **机械兜底**：本文件**无 precheck 脚本兜底** —— 决策路径性质决定无法机械化校验（边界 case 多 / 业务语境敏感）。PM 自觉对照 + Supervisor 审核兜底。

## 元规则

`[Must]` **决策路径的写法纪律**（每个新增小节必须满足）：
- 含 Q1/Q2/Q3 引导问题（按权重从高到低）
- 含各选项与适用场景对照
- 含端差异表（若与端相关）
- 含反例 / false positive（已知边界 case）
- **不写**"硬约束规则"（如 `[Must] 必须用 X`）—— 不符合本文件性质

`[Should]` **新增决策路径触发条件**：
- 同类问题在不同项目 / 调整意见中**累计出现 ≥ 3 次**
- 或产品总监明确要求沉淀

`[Recommended]` **本文件膨胀阈值**：行数超过 600 行或主题 ≥ 8 个时，按主题拆分子文件（如 `proto_data_display_list.md` / `proto_data_display_detail.md`）。

## 一、列表型选型 — 表格 vs 卡片 vs 网格 vs 瀑布流

### 选项与适用

| 选项 | 适用 | 典型 |
|------|------|------|
| **表格 Table** | 多字段 + 任务密集 + 专业用户 | 后台订单管理 / 财务流水 |
| **卡片列表 Card List** | 少字段 + 含图 + 视觉优先 | 商品推荐 / 项目卡片 |
| **网格 Grid** | 视觉对象集合（媒体 / 商品）| 图库 / 商品瀑布前置 |
| **瀑布流 Masonry** | 异构高度 + 视觉浏览 | Pinterest / 内容社区 |

### 决策路径

```
Q1 [权重最高] 同屏字段数?
   ≥ 5 字段(强对比)   → 表格
   ≤ 3 字段 + 含图    → 卡片 / 网格
   4-5 字段边界        → 进 Q2
Q2 用户主要任务?
   扫描 / 排序 / 筛选 / 批量操作 → 表格
   浏览 / 单选 / 视觉吸引         → 卡片 / 网格
Q3 数据是否含强视觉元素?
   图 / 缩略图 / 头像是核心 → 卡片 / 网格
   纯文本 + 数字 + 短状态  → 表格
Q4 用户专业度 + 频次?
   高频专业(运营 / 商户)→ 表格
   低频普通(C 端消费者)→ 卡片
```

### 端差异

| 端 | 表格字段阈值 | 推荐混合模式 |
|----|------------|------------|
| 桌面 | ≥ 5 字段不挤 | — |
| Pad 横屏 | 4 字段已偏挤 | 卡片+微表格 / 可折叠表格 |
| Pad 竖屏 | 3 字段已挤 | List + Detail 双栏 |
| 手机 / H5 | 表格难用 | 卡片优先 / 卡片+微表格 |
| 小程序 | 表格罕见 | 卡片优先 |

### 反例（已知 false positive）

- C 端银行账单流水 / 电商订单列表 → "客户端"规则会判卡片，**实际表格更合理**（多字段对比 + 排序筛选）
- B 端看板 Kanban / 媒体素材库 → "B 端"规则会判表格，**实际卡片更合理**（视觉对象 + 拖拽）
- B 端商户后台 toC SaaS → 表面是客户端，**实际是 B 端专业用户**（表格）

## 二、详情型布局选型 — 单列 / 双列 / Tab / 折叠 / 模态 / 抽屉 / 跳转

### 选项与适用

| 选项 | 适用 |
|------|------|
| **单列堆叠** | 字段少 / 移动端 / 阅读型 |
| **双列对齐** | 桌面 + 字段成对（基本信息 / 联系方式）|
| **Tab 切换** | 信息有清晰分类边界（基本 / 订单 / 操作日志）|
| **折叠面板 Accordion** | 信息层级深 + 默认折叠（设置 / FAQ 型）|
| **模态弹窗 Modal** | 主页面上下文需保持，弹出做次要事 |
| **抽屉 Drawer** | 长表单 / 详情 + 不阻断主页面 |
| **跳转新页** | 复杂主题 / 大量信息 / 独立工作流 |

### 决策路径

```
Q1 [权重最高] 用户是否需要保持原页面上下文?
   必须保留 → Modal / Drawer
   无需保留 → 单列 / 双列 / Tab / 跳转
Q2 信息总量?
   少(一屏内)        → 单列 / 双列
   中(≤ 3 屏)        → Tab / 折叠
   多(> 3 屏 / 含子流程)→ 跳转新页
Q3 信息有清晰分类边界?
   有(平级 ≥ 3 类) → Tab
   层级嵌套        → 折叠
Q4 当前是否在工作流中?
   是(向导步骤之间)→ Modal / 同页换内容
   否              → 跳转
```

### 端差异

| 端 | 模态适用 | 抽屉适用 | 跳转适用 |
|----|---------|---------|---------|
| 桌面 | 中（< 600px 宽内容）| 高（侧边详情 / 长表单）| 高 |
| Pad | 中 | 高 | 中 |
| 手机 / H5 | 低（小屏挤）| 中（底部抽屉）| 高 |
| 小程序 | 低 | 中 | 高 |

### 反例

- "编辑用户资料"含 30 字段做成模态 → **应跳转或抽屉**（模态空间不够）
- "删除确认"做成跳转新页 → **应模态**（轻量交互）
- 模态嵌套模态（用户迷失上下文）→ **应抽屉 / 拆步骤 / 改向导**
- "订单详情"含订单 / 物流 / 评价 / 售后 4 类做成折叠 → **应 Tab**（平级分类）

## 三、空 / 异常态展示选型 — empty / error / loading / no-permission

### 选项与适用

| 选项 | 适用 |
|------|------|
| **Skeleton 骨架屏** | 加载态，**已知数据结构**（列表 / 卡片骨架）|
| **Spinner 加载圈** | 加载态，**未知结构**或单次操作（提交 / 刷新）|
| **Empty State 空状态** | 无数据，含说明 + 主 CTA |
| **Error State 错误态** | 加载 / 操作失败，含原因 + 重试 |
| **No Permission 无权限** | 缺权限，含说明 + 申请入口 / 联系管理员 |
| **Placeholder Text 占位文字** | 输入框 / 极简列表项 |

### 决策路径

```
Q1 数据状态?
   loading        → 进 Q2
   empty(成功无数据)→ Empty State
   error          → Error State
   no permission  → No Permission
   ok             → 实际数据
Q2 加载场景?
   首次列表 / 详情加载(已知结构)→ Skeleton(比 Spinner 体验好,无跳变感)
   操作中(提交 / 请求)         → Spinner / Toast 进度
   无限滚动加载更多            → 底部 Spinner + 文案
```

### 端差异

跨端通用，但手机端更强调"骨架屏减少跳变感"（小屏对内容抖动更敏感）。

### 反例

- 列表首次加载用 Spinner 不用 Skeleton → 用户看不见"将要呈现什么结构"
- Empty State 仅展示"暂无数据" 无 CTA → 用户不知下一步
- Error 仅展示"出错了" 无原因 / 无重试按钮 → 用户卡死
- 把"无权限"做成"无数据"（误导，用户以为系统空）
- 加载 + empty 混淆（loading 转 empty 中间无过渡，闪烁）

## 四、海量数据展示选型 — 分页 / 虚拟滚动 / 加载更多 / 流式

### 选项与适用

| 选项 | 适用 |
|------|------|
| **分页 Pagination** | 总数已知 + 用户需跳页 / 直达末页 |
| **加载更多 Load More** | 用户需手动控制 + 视觉上保留已读 |
| **虚拟滚动 Virtual Scroll** | 大量（> 1000）+ 同结构 + 顺序浏览 |
| **流式 Streaming** | 实时推送 / 日志型 / 消息流 |

### 决策路径

```
Q1 [权重最高] 数据总量?
   ≤ 50 条                  → 一次性加载,无需分页
   50-1000 同结构           → 分页 / 加载更多
   > 1000 同结构 + 顺序浏览 → 虚拟滚动
   实时增量推送             → 流式
Q2 用户是否需要"跳转到 N 页 / 直接看最后页"?
   是(后台数据管理 / 工单)→ 分页
   否(浏览型 / 推荐流)    → 加载更多 / 虚拟滚动
Q3 端类型?
   桌面 / Pad → 分页 / 虚拟滚动均 OK
   手机 / 小程序 → 加载更多优先(分页按钮触控难)
Q4 是否需要"保留滚动位置"(返回时回到原位)?
   是 → 加载更多 / 虚拟滚动(分页会丢位置)
   否 → 分页可
```

### 端差异

| 端 | 分页 | 加载更多 | 虚拟滚动 |
|----|------|---------|---------|
| 桌面 | 高频 | 中 | 高频（大列表）|
| Pad | 中 | 高 | 中 |
| 手机 / H5 | 低（按钮难点）| 高频 | 中（性能需关注）|
| 小程序 | 低 | 高频 | 罕见（框架限制）|

### 反例

- 客户端朋友圈 / 推荐流用分页 → **应加载更多 / 无限滚动**（沉浸式被打断）
- 后台日志 10 万条用加载更多 → **应虚拟滚动 + 跳转**（手动加载太累）
- 商品列表 10 万条用分页 → **应搜索 / 筛选 + 虚拟滚动**（用户翻不到）
- IM 消息列表用分页 → **应流式 + 历史加载更多**（实时性需求）

## 五、主列表 + 辅助功能区布局选型 — 左右 vs 上下

### 模式定义

主体是数据列表（商品 / 报价 / 项目 / 文件），辅助区是与主体强耦合的**操作 + 汇总**（价格 / 优惠 / CTA / 详情预览 / 属性面板）。**关键特征**：辅助区**不是次要列表**，而是"基于主体内容的派生信息或操作"——区别于"双列列表"（双列列表参见 §一 网格 / §二 List+Detail）。

### 选项与适用

| 选项 | 适用 | 典型场景 |
|------|------|---------|
| **左右布局 主 + Aside** | 桌面 / Pad 横屏 + 主体宽度允许 | 购物车 / 报价清单 / 设计工具 / 邮件 |
| **上下布局 主 + 底部栏** | 主体宽度撑满 / 移动端 | 复杂表单 / 多字段列表 / 手机商品列表 |
| **左右布局 + 折叠辅助区** | Pad 竖屏 + 辅助区临时可隐 | 文件管理器（隐藏预览）/ 报表（隐藏筛选）|
| **抽屉 / 模态替代** | 辅助区仅在特定操作时呼出 | 添加优惠券 / 修改地址 |

### 决策路径

```
Q1 [权重最高] 主体列表的宽度需求?
   主体需 ≥ 70% 屏宽(多字段表格 / 宽卡片)→ 上下布局
   主体 ≤ 60% 屏宽(单字段 + 缩略图 / 紧凑卡片)→ 进 Q2
   边界 60-70% → 进 Q2 + 评估辅助区可压缩度

Q2 辅助区与主体的耦合强度?
   强耦合(主体选择实时影响辅助区 / 辅助区 CTA 作用于主体全集)
       → 左右价值高(辅助区 sticky,操作触手可及)
   弱耦合(辅助区是独立信息)
       → 左右收益低,考虑上下或抽屉

Q3 辅助区性质决定其宽度?
   汇总 + 单 CTA(购物车结算 / 报价确认)  → 窄 sidebar 25-30%
   操作面板(参数调节 / 多按钮)            → 中 sidebar 30-35%
   详情阅读区(选中项详情)                  → 大 sidebar 40-45%
   不可压缩 → 考虑改用 Tab / 跳转,而非左右

Q4 主体是否需要独立滚动?
   长列表(> 一屏)且辅助区需常驻可见 → 左右 + 辅助区 sticky 是最佳解
   短列表(≤ 一屏)                  → 上下也 OK,左右非必须

Q5 端类型?
   桌面(≥ 1024px)→ 左右优先
   Pad 横屏      → 左右,辅助区窄(25-30%)
   Pad 竖屏      → 多数退化为上下 / 折叠辅助区
   手机 / H5 / 小程序 → 上下 + 吸底 CTA 栏(辅助信息退化为底部摘要条)
```

### 推荐布局比例

| 场景 | 主 : 辅 | 备注 |
|------|--------|------|
| 购物车 / 报价清单（汇总 + CTA）| 70 : 30 | 辅助区固定窄，sticky |
| 设计工具（属性面板）| 70 : 25（+ 左侧 5% 工具栏）| 三栏 |
| 邮件 / 文件管理（详情）| 50 : 50 / 40 : 60 | 详情区主导 |
| 数据列表 + 筛选侧栏 | 75 : 25 | 筛选区可折叠 |

### 端差异

| 端 | 推荐布局 | 主体列表形式 | 辅助区位置 |
|----|---------|------------|-----------|
| 桌面 | 左右 70:30 | 表格 / 紧凑卡片 | 右侧 sticky |
| Pad 横屏 | 左右 65:35 | 卡片 + 微表格 | 右侧 sticky |
| Pad 竖屏 | 上下 / 折叠辅助 | 卡片 | 顶部摘要条 + 底部 CTA / 可展开抽屉 |
| 手机 / H5 | 上下 | 卡片 | 底部吸底栏（金额 + CTA）|
| 小程序 | 上下 | 卡片 | 底部吸底栏 |

### 反例（已知 false positive）

| 错配 | 典型表现 | 应改为 |
|------|---------|--------|
| 商品列表 5+ 字段强对比仍坚持左右 | 表格列被挤变形 | **上下**（主体撑满）|
| 辅助区 CTA 单一 + 内容少（仅"立即下单"）| 占整侧栏浪费 | **吸底 CTA 栏** / 上下 |
| 强工作流（不能离开当前主任务）| 左右布局上下文反而割裂 | **抽屉**（呼出辅助）|
| Pad 竖屏强行左右 65:35 | 辅助区挤到 < 200px 不可用 | **上下** / 折叠辅助 |
| 辅助区有大量编辑表单（> 8 字段）| 侧栏空间不够 | **改抽屉 / 跳转编辑页** |
| 主辅都需独立滚动 + 都很长 | 用户找不到锚点 | **重新评估信息架构**（拆页面）|

### 典型案例参考

**购物车（C 端电商）**：
- 桌面 / Pad 横屏：左右 **70:30**，主体紧凑卡片（图 + 标题 + 单价 + 数量 + 小计），右侧 sticky 结算（小计 + 优惠 + 配送 + 大主 CTA "去结算"）
- Pad 竖屏 / 手机：上下，结算栏吸底（总金额 + "去结算"）

**报价清单（B 端专业工具）**：
- 桌面 / Pad 横屏：左右 **65:35**（主体稍宽，含规格 / 单价 / 数量 / 小计多字段），主体用**表格**（B 端专业用户多字段对比），右侧 sticky 汇总（小计 / 优惠 / 总价 + 客户备注 + "保存草稿" + "确定报价"）
- Pad 竖屏：折叠右侧汇总或上下
- 手机：上下 + 底部吸底栏（总价 + "确定报价"）

## 反 pattern（常见错配速查）

| 错配 | 典型表现 | 应改为 |
|------|---------|--------|
| 表格里塞图片缩略图作为主视觉 | 用户点不准列 | 网格 / 卡片 |
| 卡片列表用于专业高频任务 | 滚动远 + 字段对比难 | 表格 |
| 模态嵌套模态 | 用户迷失上下文 | 抽屉 / 拆步骤 |
| Empty State 仅"暂无数据" | 用户不知下一步 | 加 CTA / 入口 |
| 移动端用全功能表格 | 横向滚 + 点不准 | 卡片 / 折叠表格 |
| 首屏列表用 Spinner | 内容跳变感强 | Skeleton 骨架屏 |
| 删除确认用跳转新页 | 操作链路过长 | 模态 |
| 10K+ 条数据用分页 | 用户翻不到 | 虚拟滚动 + 搜索筛选 |
| Tab 项 ≥ 7 | 用户找不到目标 | 折叠 / 二级菜单 |
| 折叠面板默认全展开 | 失去折叠意义 | 默认折叠首屏外 |
