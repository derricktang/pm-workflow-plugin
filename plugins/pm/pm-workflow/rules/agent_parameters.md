# Agent 参数声明 v1.0

> **本文件定位**：`agent_methodology.md` 的实例化伙伴文件——前者定义通用方法论与参数槽位，本文件填入本工作流各 Agent 在每个槽位上的具体值。
> **维护原则**：方法论新增/调整槽位时同步更新本文件；新增 Agent 类型时在本文件矩阵增加一列，参数槽位本身不在本文件定义。
> **使用方式**：subagent 派发时同时传入 `agent_methodology.md` + 本文件，subagent 按方法论遵循义务执行，按本文件参数实例化具体动作。

---

## 一、本工作流当前 Agent 列表

| Agent 类型 | 职责 | 主体动作 |
|-----------|------|---------|
| PM Agent | 产出方 | 起草各阶段成果文档 |
| Supervisor Agent | 审核方 | 审核 PM 产出并出具审核结论 |

未来扩展角色（UI / 测试 / 开发 Agent 等）按文末「新增角色指南」流程接入。

## 三、角色 atomic step 声明表

> **本节定位**：各角色为 T1-T6 每个时序条目声明其 atomic step 切分清单及验收点。切分必须满足 `agent_methodology.md` 的 X1 配套四条判据。当某角色对某条目的执行流程发生重大调整时，须同步更新本节。
>
> **[Must] SSOT 三方双向链**：本表 `tN.M` 子步骤是基于 `agent_methodology.md §二·T1-T6 各节奏「参数化维度」+ §三·X1 配套通用切分判据`双重授权的合法实例化（**非参数文件越权造概念**）。三方关系：
>
> - 方法论 `§二·各节奏` —— 定义 T1-T6 不变式 + 参数化维度（含"子步骤切分"维度，授权下放）
> - 方法论 `§三·X1 配套切分判据` —— 定义切分四条判据（可独立验收 / 可独立重做 / 产出可追溯 / 边界贴合不确定性）
> - 参数声明 `§三 本节` —— 各角色实例化具体 `tN.M` 子步骤清单 + 验收点
>
> 任何切分调整须先核对方法论双重授权约束未被违反，再改本表；新角色（如自动化测试 Agent）复用 T1-T6 框架时直接在本节增加 `§3.X 角色名 atomic step 声明`，命名约定沿用 `tN.M`，不得自造私有体系。
>
> **草稿状态说明**：以下 PM / Supervisor 的 atomic step 清单为方法论起草阶段的初稿，落地时须由各 Agent 文档维护方对照实际工作流核实并细化。
>
> **回溯路径说明**：各 atomic step 失败时的回溯目标见 `agent_methodology.md` §二·T6「失败处理」段（按问题属性映射）+ §三·X3「失败处理」段（按 step 来源映射）。

- **分支步数表达约定**：当子角色含互斥分支（如项目组件识别的 PI3-PI6 与 PI7）时，T5 步数列展示**总步数 + 各分支实际执行步数**，格式 `N（分支：A 路径 X 步 / B 路径 Y 步）`。Step 1.X 主管中段审核须按 PM 进度文件中"分支判定"行匹配的分支步数验收，不按总步数验收；不执行的分支步骤在进度文件标 `[N/A]`，详见 `AI产品经理_Agent.md §五 模块进度并发纪律` N/A 标记规则。

**[Must] W 步范围界定**：本矩阵 W 步前缀**仅含 PM 自己执行**的 atomic step（PM 在主进度文件 `stage4_[产品名]_plan.md` 上勾选 `[x]` 的步骤）。主文件中其他**编排器执行**的步骤标记**不纳入** W 步前缀、不属于任何 PM 子角色的 atomic step、无需 X1 配套 4 条判据自检——这些步骤由编排器单线程勾选，仅作为工作流可追溯性的占位。当前已识别的编排器执行步骤标记：

| 主文件标记 | 执行者 | 命令 |
|-----------|-------|------|
| `TP5` | 编排器 | `python pm-workflow/scripts/gen_scaffold.py` |
| `AS` | 编排器 | `python pm-workflow/scripts/assemble.py spec` |
| `AP` | 编排器 | `python pm-workflow/scripts/assemble.py prd` |
| Step 6.5（无 W 步前缀）| 编排器 | `python pm-workflow/scripts/precheck_stage4.py` |

未来若新增编排器执行步骤，须同步追加本表，**不得**纳入 PM 子角色 W 步前缀（否则 PM X1 自审与 Step 1.X 主管审核会因"PM 无法对编排器步骤独立验收"卡住）。

**[Must] X1 配套切分判据复核**：本矩阵中每个子角色的 T5 W 步前缀清单（**仅 PM 执行的步骤**）都已通过 `agent_methodology.md §三·X1 配套通用切分判据` 4 条（可中断 / 可独立重做 / 可独立完成验收 / 可回溯）的声明性验证；具体步骤定义见 `AI产品经理_Agent.md §五`。若工作流调整后某子角色的 W 步清单无法满足 4 条判据，须先在本矩阵升级前缀定义、再调整 §五。

## 四、新增角色指南

新增 Agent 类型（如 UI / 测试 / 开发 Agent）的接入流程：

1. **方法论合规性检查**：通读 `agent_methodology.md`，确认本角色的工作流可以由 T1-T6 + X1-X4 + 元规则覆盖；如发现某不变式不适用，说明需要扩展方法论而非新增角色——升级为方法论变更议题
2. **填齐主参数矩阵**：在 §二「主参数矩阵」增加一列，对照每一行参数槽位填入本角色的具体值；空缺槽位须显式标注 "—" 或说明原因
3. **声明 atomic step 清单**：在 §三新增 "3.N 角色名 atomic step 声明" 子节，按 T1-T6 顺序列出本角色的切分；每个 atomic step 必须通过 X1 配套四条判据自检
4. **更新本文件 §一 角色列表**：追加新角色及其职责说明
5. **写本角色主文档**：新角色的角色规范、阶段任务、自审清单等角色特有内容写在该角色独立的 `AI_角色名_Agent.md` 中，文档头部加方法论遵循声明 + 链接本文件

**禁止事项**：
- 不得修改 `agent_methodology.md` 的任何不变式条目（方法论是不可变层）
- 不得在角色主文档中重复定义已在本文件声明的参数（避免单点维护破坏）
- 不得跳过 X1 配套四条判据自检直接接入 atomic step 清单
